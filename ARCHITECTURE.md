# Pulse_Brain — Architecture

This document defines how Pulse_Brain is designed and the order in which it is built. It is the single source of truth for the system's shape; each section updates it as the brain grows.

## 1. Engine vs instance

Pulse_Brain exists in two tiers with a hard boundary between them.

- **Engine — `Pulse_Brain`** (this repo). Architecture only: directory structure, operating rules, frontmatter templates, skill and agent definitions, documentation. It is sanitised by design and safe to version in git. It is the reusable model from which any number of instances can be created.
- **Instance — `Pulse-{{OWNER}}-Brain`** (created in a later section). A live Obsidian vault on the machine, instantiated from the engine, holding real {{COMPANY}} knowledge. Private, local, never committed.

**The data boundary is absolute.** Live data, secrets, client names, financial figures, and PII never enter the engine repo. The engine ships *templates and structure*; the instance holds *content*. This boundary is enforced by `.gitignore` and by the security protocol in `.claude/CLAUDE.md`.

**Roles.** Within the project, work runs on a **dual-role model** governed by the project director (`.claude/CLAUDE.md`): **Pulse** is the brain (memory, knowledge, identity, curation, growth) and **Claude** is the mind (reasoning, execution, agent orchestration). The full definition and the collaboration seam live in `.claude/CLAUDE.md` — the authority for roles.

## 2. Knowledge model — a compiled wiki over raw sources

**Compile, don't retrieve.** `canon/` is a **living wiki** the LLM maintains; `inbox/` and `dynamic/` are the raw **sources** compiled into it. Knowledge is compiled once and kept current, not re-derived per query.

- **`canon/` — the living wiki.** Entity pages, topic summaries, the compiled synthesis. The LLM **maintains it freely** and **flags** contradictions/major revisions for review — there is **no pre-write confirmation gate** (this supersedes the earlier S3/S4 "locked / gated" model).
- **`inbox/` — {{OWNER}}'s sources.** Files, URLs, pages, thoughts {{OWNER}} adds.
- **`dynamic/` — agents' sources.** Research and monitoring agents gather. 

`inbox/` + `dynamic/` are raw inputs by origin, kept for provenance; `canon/` is the synthesis they feed. Each pillar is organised by the dual-role **domains** (`gtm`, `ai-orchestration`, `product`, `people`, `operating`, `glossary`; plus `entities`, a canon-only wiki page type).

**Frontmatter contract.** Every entry carries at least a `title` and a one-sentence `description`. `[[wikilinks]]` cross pillar boundaries freely so the whole brain renders as one connected graph in Obsidian. The skeleton lives in [`vault-template/`](vault-template/); the full taxonomy, per-pillar frontmatter contracts, and lifecycle are specified in [`TAXONOMY.md`](TAXONOMY.md). The pillars are the **Context Engine** layer of the operating system (below).

## 3. The Brain Operating System

The brain is a **layered, living system** — not a filing cabinet. Its operating system is in [`BRAIN-OS.md`](BRAIN-OS.md): **ten layers** (Operations Kernel · Context Engine · Thinking · Research & Background Agents · Builder · Self-Development & Growth · Persona/Presets · Development · Interface · **Model & Retrieval Substrate**) over **three laws** — *living vault* (every input runs the integration pipeline: ingest → integrate into the wiki → link → re-index), *atomic notes* (one idea per file; frontmatter → summary → body; `[[wikilinks]]`), and *intelligence partner* (thinks *with* {{OWNER}} — proactive, honest, advising). Its runtime is the **Operations Kernel** ([`OPERATIONS-KERNEL.md`](OPERATIONS-KERNEL.md)), which exposes a **capability contract** that skills/agents/commands are built against. The **Context Engine** layer — the knowledge substrate (`canon/` ◀ compile ─ `inbox/`/`dynamic/`) and the editorial standard for it — is specified in [`CONTEXT-ENGINE.md`](CONTEXT-ENGINE.md). The **node/edge model** the whole brain runs on — what its nodes and edges are, how they are traversed, judged, and grown — is the authority in [`NODE-GRAPH.md`](NODE-GRAPH.md).

The brain **develops by growing its node-graph** — relational learning, with the frontier model pluggable (Claude now, others later); the owned graph is the moat. Self-growth is two layers: **Builder** (a capability for a present problem) and **Self-Development & Growth** (the meta-loop evolving the platform). Each layer is built in its own section (roadmap below). The brain is **instruction-first**; code enters only where a layer needs a process, UI, or hard determinism (see the instruction-vs-code map in `BRAIN-OS.md`).

## 4. Cross-surface access

Pulse_Brain is operated **exclusively through Claude Code on the CLI and desktop app — primarily the CLI.** The intended working pattern is **multiple concurrent sessions**: several tabs open at once, each running one or more agents that work on {{OWNER}}'s behalf against the same shared local brain. There is no dependency on the web/claude.ai surface or any external connector.

## 5. Section roadmap

Pulse_Brain is built one planned section at a time. This list is a living plan — refined as each section completes.

- **S1 — Foundation** *(complete)*. Engine repo, identity, architecture, project director skeleton, security boundary.
- **S2 — Operating identity & standards** *(complete)*. Dual-role model defined (Pulse=brain, Claude=mind); tone, structure, and quality/security standards built out in `.claude/CLAUDE.md` as the project's top authority for roles.
- **S3 — Vault taxonomy & frontmatter templates** *(complete)*. Three-pillar origin-based layout (`canon`/`dynamic`/`inbox`) with dual-role domains in `vault-template/`; five frontmatter templates; full spec in `TAXONOMY.md`.
- **S4 — Brain OS architecture (generic)** *(complete)*. The layered operating system and three laws defined generically in `BRAIN-OS.md`; templates upgraded to the atomic-note standard. Each layer/pillar then built in its own section:
- **S5 — Operations Kernel — the wiki engine** *(complete)*. Compile-don't-retrieve: ingestion front-door, integration pipeline, graph-first read, edge-map index, `index.md`/`log.md`, two-tier memory, capability contract, model-pluggability. Spec in `OPERATIONS-KERNEL.md`.
- **S6 — Context Engine: Canon (the wiki)** *(complete)*. The editorial standard for the wiki: the three page kinds, atomicity & page boundaries, maintain-freely revision, the flag-don't-gate contradiction protocol with resolution lifecycle, provenance, page-level graph discipline, page lifecycle, and the quality bar. Spec in `CONTEXT-ENGINE.md` (Canon section); S7/S8 extend the same doc.
- **S7 — Context Engine: Dynamic sources** *(complete)*. The agent-gathered source pillar: the capture contract a well-formed source meets, a structured `confidence` (high/medium/low) trust model for open-world reliability, and the capture → integrate → feed lifecycle. Spec in `CONTEXT-ENGINE.md` (Dynamic section); `confidence` added to the dynamic-note contract in `TAXONOMY.md`.
- **S8 — Context Engine: Inbox sources** *(complete)*. The {{OWNER}}-gathered source pillar: two integration sub-kinds (primary thoughts/decisions vs curated-external material), source modalities including image/video (multimodal), the capture contract, and frictionless capture ergonomics on the CLI channel. Spec in `CONTEXT-ENGINE.md` (Inbox section). **Completes the Context Engine layer** (Canon + Dynamic + Inbox).
- **S9 — Node-Graph** *(complete)*. The node/edge model the whole brain runs on, consolidated into one authority (`NODE-GRAPH.md`): node kinds, structural edges (`feeds`/`sources`/`related`/`[[wikilink]]`), a **seeded-but-open** semantic-edge model (`supersedes`/`contradicts`/`corroborates` + a promotion rule so the ontology grows with the brain), traversal, graph-health signals, and growth rules. Instruction-tier; the code substrate that runs it is S17.
- **S10 — Thinking layer** *(complete)*. The brain's first **active** layer (reasons over the graph rather than maintaining it). Five operations — synthesis pages, link proposal, contradiction *resolution proposal* (feeds Canon §4), content-promotion candidates (ontology promotion stays S13's), and challenge & advise (Law 3) — built on the kernel contract with no new primitive. Anchored by the **inference-marking** discipline: the `[!inference]` callout (third face of *unverified claims are always marked*, with `confidence` and `[!contradiction]`), confidence-aware auto-resolution, the S11 self-correction loop, and a `log.md` audit stamp for autonomous confirmations. Spec in `THINKING.md`.
- **S11 — Research & Background Agents layer** *(complete)*. The brain's first **code** layer (Claude Code operating artifacts, not a bespoke daemon): two worker kinds (research/pull, monitor/cadence), the gather lifecycle, the inference-queue drain that closes the S10 loop, and the scheduling model (live cron + watchlist deferred to instance time as content). Built under the five governing principles (Agent-First · Test-Driven · Security-First · Immutability · Plan-Before-Execute). Spec in `RESEARCH-AGENTS.md`; machinery in `vault-template/.claude/` (skills · agents · commands · tests · `instruments.md` registry · scheduling template). **Two instruments brought in alongside, in their own lanes:** a **Builder seed** (`create-skill`/`create-agent`/`create-command`, wrapping the global `create` — a minimal S12 primitive pulled forward, the extensibility ground) and an **`agent-architecture-audit`** QA instrument (12-layer agent-stack diagnostic). Scope: a minimal exemplary set; the population grows on real need via S12/S13.
- **S12 — Builder layer** *(complete)*. Reactive self-extension: when a task needs a capability that does not exist, the brain builds one instrument to solve it — against the kernel contract, registered, and audited. Generalises the S11 seed into one **build lifecycle** over **four instrument kinds** (skill · agent · command · **tool**, the one code kind — a deterministic helper script), a unified `/build` entry, the **reactive gap-detection trigger** with a **dual-role build gate** (Claude detects, Pulse decides and records), and a Test-Driven acceptance bar. Authority in `BUILDER.md`; the fourth executor (`create-tool`) and `/build` ship in `vault-template/.claude/`. Heavy code substrate (MCP host, embeddings) stays S17/S18; the meta-loop that *drives* Builder is S13.
- **S13 — Self-Development & Growth layer** *(complete)*. The meta-loop that grows the **platform itself**: it audits the brain's own condition and **drives** Builder/Research/Thinking to mend it — it is the judgement that points the hands, never the hands (the sharp line from S12, which is reactive single-instrument creation). One bounded loop — **audit → triage → route → drive → record** — over the seven [`NODE-GRAPH.md`](NODE-GRAPH.md) §6 graph-health signals (each routed to the layer that owns the fix) plus the two growth decisions S13 **owns**: ontology promotion (§3.3) and systemic capability gaps. Anchored by a **three-tier autonomy gate** keyed on *additive-and-isolated vs mutating-shared-meaning*: compute/route is free, driving Builder inherits build-then-record, and changes to shared meaning (ontology / architecture) are **propose-build-complete → ratify** — the brain never silently rewrites its own engine. Instruction-tier; the code that computes health metrics at scale is S17. Spec in `SELF-DEVELOPMENT.md`; machinery in `vault-template/.claude/` — `self-develop` skill, `growth-agent`, `/grow`, the **engine's first `tool`** (`graph-health-scan`, dogfooding S12's `create-tool`), and the `growth-proposal-contract` acceptance bar.
- **S14 — Persona / Presets layer** *(complete)*. Selectable operating **presets** for {{OWNER}}'s roles — a preset is a **lens, not a store**: it biases *focus, default domains, tone, and which instruments lead* for one role (GTM Lead, AI Orchestrator, others) **without changing the knowledge, the identity, or the laws beneath it**. Instruction (config) tier; **no new primitive** — switching sets the `active_preset` anchor the kernel already reads ([OPERATIONS-KERNEL §7/§10](OPERATIONS-KERNEL.md)) via `update-memory`. Anchored by two boundaries: **lens-not-filter** (default domains *lead* read-routing, never suppress a domain the task needs or hide a `core.md` flag) and **one voice** (a preset is a scoped operating role beneath the director — `tone` is register/emphasis, never a second identity). Spec in `PERSONA-PRESETS.md`; ships the preset contract (`_templates/preset.md`), two exemplar presets (`presets/gtm-lead.md`, `presets/ai-orchestrator.md`), the `/preset` command, and the `preset-contract` acceptance bar.
- **S15 — Development layer** *(complete)*. Multi-tab / concurrent-session operation over one shared local vault, plus the workflow for developing the brain itself. Local-filesystem coordination, not distributed consensus: concurrency is made **safe by construction and self-healing, not prevented by locking**. Anchored by a **write-class taxonomy** (append-only/new-file is collision-free; `core.md` is a reconcilable cache; co-edited shared meaning is kept rare by lane discipline and the session note) and the **ledger guarantee** — `log.md` is the append-only truth, `core.md` a cache the kernel rebuilds from it, so a clobber is always recoverable. Instruction (+ light tooling): ships the `sessions.md` append-only presence note and one deterministic `session-reconcile` tool (compacts the note, reaps crashed tabs, reports `core.md` drift; writes only its own file, repairs nothing it does not own), on two triggers — a `SessionEnd` hook (clean exit) and a loose ~15' backstop (crash). The reconciler **heals, it does not prevent**: write-time leasing for true prevention, automated conflict detection, and merge arbitration are the named **deferred broad scope**. Spec in `DEVELOPMENT.md`; machinery in `vault-template/` (`memory/sessions.md`, `.claude/tools/session-reconcile`, the `development-contract` acceptance bar).
- **S16 — Interface layer** *(in progress — boot/entry shipped)*. The UX surfaces through which {{OWNER}} enters, feeds, and reviews the brain ([`INTERFACE.md`](INTERFACE.md)), built **lean and design-first** — designed in full, realised where honest against the local vault, with the experiential parts deferred to the live vault (S19). **Boot/entry shipped:** the `/pulse` command wakes the brain (paints a byte-exact banner, runs read-routing, greets {{OWNER}}, reports preset · focus · review · last) — summoned deliberately, not auto-booted. **Capture is channel-agnostic, no bespoke app** (decided with {{OWNER}}): CLI (S8) + Remote Control for the phone with the Mac awake (local-only, does not cross the data boundary) + a synced drop-folder for async capture (the sync provider is the always-on intermediary — no app, no backend), with a hosted endpoint deferred to S18. The Obsidian review surface stays lean convention now and matures at S19. The mobile-capture-app idea is **superseded** by Remote Control + drop-folder.
- **S17 — Model & Retrieval Substrate** *(complete — the deterministic shell; calibration + read-routing integration at S19)*. The ML engine room beneath the kernel ([`MODEL-RETRIEVAL.md`](MODEL-RETRIEVAL.md)), realising two kernel hooks: **`model-route`** (the deterministic `resolve-model` tool + an instance bindings template + the documented Opus-loop / tiered-worker policy and its cache-the-prefix, tier-the-workers cost rationale) and the **semantic tie-breaker** (an embedding substrate that **re-ranks** the graph walk's candidates — never retrieves graph-distant pages, so compile-don't-retrieve holds). Introduces the **`substrate/`** code category (engine code beneath the kernel — it uses the network/credentials, so it is not a Builder `tool`; admitted by an `instruments.md` §1 carve-out). Built lean: the shell ships (embedding interface + local/API adapters + a deterministic stub; a per-page-shard derived-cache store with a read-time staleness guard + atomic single-writer rebuild; candidates-only dedup/edge finders; pinned-model + checksum-on-pull). Deferred to S19: threshold calibration, the real model pull, and turning re-ranking on in read-routing. The metered self-tuning routing loop was **cut** (its telemetry is uncapturable at this tier); MCP hosting is **S18**.
- **S18 — MCP hosting.** Expose Pulse as an external MCP server (code). Also the **sync backend** the mobile capture app (S16) needs to relay captures back into the local brain.
- **S19 — Instantiation kit** *(the distribution layer; reframed from a one-off "instantiate the owner's vault" into a reusable, stranger-proof kit, dogfooded by the owner's own first instance)*. A clone of the engine becomes a working private brain via **`/setup`**: a `setup` skill interviews the owner (identity, **owner-defined domains**, presets, writing voice, embedding choice) and drives the deterministic **`instantiate`** tool, which stamps a live instance **outside the engine** (Model A — the instance preserves the engine's nesting so every instrument→doc link resolves; authority docs copied unstamped, operational surfaces stamped, owner domains scaffolded, empty state seeded, no-placeholder verified, **refuses any target inside the engine**). Procedure in [`SETUP.md`](SETUP.md); the data boundary is safe by construction (the populated instance is a separate dir, never committed). Shipped: `instantiate`, `/setup` + the `setup` skill, `SETUP.md`, `.env.example`, the `instantiation-contract` bar, and the genericity edits (domains/presets are owner-defined examples).
- **S20 — Go public / sharing** *(in progress)*. A newcomer-facing README and the **MIT `LICENSE`** (copyright the author — see [`LICENSE`](LICENSE)) ship with this section; the engine goes public via a **fresh clean-history snapshot** (the full-history private repo is retained as a backup; the public repo begins from one clean commit, so no real-name/email history travels). The private remote was established early (after S9) as off-machine backup, safe because the engine is sanitised by design. Remaining: the public flip itself, and any later sharing/collaboration decisions.

## 6. Conventions

- **Language:** British English.
- **Markdown + frontmatter:** all knowledge is plain markdown with YAML frontmatter, Obsidian-compatible.
- **Linking:** `[[wikilinks]]` for cross-references.
- **Security:** see `.claude/CLAUDE.md` — the engine/instance data boundary is a hard rule.
- **Placeholders:** the engine is generic and names no real person or company. Two placeholders stand in for instance-specific identity and are filled when the living vault is instantiated (S19):
  - `{{OWNER}}` — the brain's owner.
  - `{{COMPANY}}` — the owner's organisation.
  - The owner's instance follows the pattern `Pulse-{{OWNER}}-Brain`. Auditing the engine for leaks is therefore a grep for real names returning nothing, and a grep for `{{` returning only intended template markers.
